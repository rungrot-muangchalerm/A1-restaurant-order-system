# Requirement เว็บไซต์ร้านอาหารและระบบร้านอาหารแบบ Full-System

## 1. วัตถุประสงค์ของระบบ
1. ใช้เป็นเว็บไซต์ร้านอาหารออนไลน์ที่ลูกค้าเข้ามาแล้วเห็นภาพลักษณ์ร้าน เมนูอาหาร โปรโมชัน ช่องทางติดต่อ และสามารถเริ่มสั่งอาหารได้ทันที
2. ใช้เป็นระบบร้านอาหารครบวงจร ตั้งแต่ลูกค้าดูเมนู สั่งอาหาร ติดตามสถานะ รับอาหาร ชำระเงิน ไปจนถึงการดูประวัติคำสั่งซื้อ
3. ใช้เป็นระบบ POS สำหรับพนักงานหน้าร้านในการเปิดออเดอร์ เพิ่มรายการ แก้ไขออเดอร์ รับชำระเงิน และปิดบิล
4. ใช้เป็นระบบครัวสำหรับรับรายการอาหารจากทุกช่องทาง และอัปเดตสถานะการปรุงอาหารแบบเป็นลำดับ
5. ใช้เป็นระบบบริหารหลังร้านสำหรับจัดการเมนู ราคา โต๊ะ ผู้ใช้ สิทธิ์ โปรโมชั่น รายงาน และประวัติการทำรายการ
6. ใช้สรุปยอดขาย เมนูขายดี จำนวนออเดอร์ วิธีชำระเงิน สถานะบริการ และประสิทธิภาพการทำงานของร้าน

## 2. ภาพรวมของระบบ
ระบบต้องเป็น `one port restaurant website` คือผู้ใช้ทุกคนเข้ามาที่เว็บเดียวกัน เห็นหน้าเว็บร้านอาหารเดียวกัน และใช้ navbar ชุดเดียวกัน

หน้าแรกของระบบคือหน้าเว็บไซต์ร้านอาหาร ไม่ใช่หน้าเอกสาร ไม่ใช่หน้า dashboard และไม่ใช่หน้า shortcut สำหรับเข้า POS หรือ admin

ระบบแบ่งการทำงานออกเป็น 6 ส่วนหลัก

1. `public restaurant website`
หน้าแรกของร้าน แสดงบรรยากาศร้าน เมนูแนะนำ โปรโมชัน เรื่องราวของร้าน แกลเลอรี รีวิวลูกค้า วิธีสั่งอาหาร และข้อมูลติดต่อ

2. `customer system`
ลูกค้าดูเมนู เพิ่มสินค้าในตะกร้า สั่งอาหาร ระบุโต๊ะหรือประเภทการรับอาหาร ติดตามสถานะ และชำระเงินตามรูปแบบที่ร้านกำหนด

3. `staff / POS system`
พนักงานเปิดออเดอร์แทนลูกค้า จัดการโต๊ะ เพิ่มหรือแก้ไขรายการ ส่งออเดอร์เข้าครัว และช่วยปิดบิล

4. `kitchen system`
ครัวเห็นรายการอาหารที่ต้องปรุง แยกตามคิว เวลา โต๊ะ ประเภทการสั่ง หมายเหตุ และเปลี่ยนสถานะอาหาร

5. `cashier system`
แคชเชียร์ตรวจสอบยอด รับชำระเงิน บันทึกวิธีชำระ ออกใบเสร็จ และปิดบิล

6. `admin / back office system`
ผู้ดูแลระบบจัดการข้อมูลหลัก ผู้ใช้ สิทธิ์ เมนู ราคา โต๊ะ โปรโมชั่น รายงาน และตรวจสอบประวัติการใช้งาน

## 3. ขอบเขตของระบบ
1. รองรับการใช้งานจากลูกค้า พนักงาน ครัว แคชเชียร์ ผู้จัดการ เจ้าของร้าน และผู้ดูแลระบบ
2. รองรับประเภทการสั่งอย่างน้อย ได้แก่ ทานที่ร้าน กลับบ้าน และเดลิเวอรี
3. รองรับการสั่งอาหารด้วย QR Code สำหรับโต๊ะอาหาร หรือเปิดออเดอร์โดยพนักงาน
4. รองรับการจัดการเมนู หมวดหมู่ รูปภาพ ราคา สถานะพร้อมขาย และตัวเลือกเพิ่มเติมของเมนู
5. รองรับการจัดการโต๊ะ โซน จำนวนที่นั่ง สถานะโต๊ะ และ QR Code ประจำโต๊ะ
6. รองรับการส่งรายการอาหารเข้าครัว อัปเดตสถานะ เสิร์ฟอาหาร และปิดบิล
7. รองรับการชำระเงินหลายวิธี เช่น เงินสด โอน บัตร หรือ QR Payment ตามที่ร้านกำหนด
8. รองรับรายงานยอดขาย รายงานเมนูขายดี รายงานออเดอร์ รายงานการชำระเงิน และประวัติการแก้ไข
9. ระบบในเอกสารนี้สามารถนำไปทำเป็น mockup HTML ก่อน โดยยังไม่ต้องมีฐานข้อมูลหรือระบบจริง
10. หน้าแรกต้องให้ความรู้สึกเป็นเว็บไซต์ร้านอาหารจริง มีภาพอาหารหรือบรรยากาศร้าน มีเมนูแนะนำ โปรโมชัน รีวิว และข้อมูลติดต่อ
11. ระบบหลังบ้าน เช่น POS, kitchen, cashier และ admin ต้องเป็นส่วนที่เข้าถึงจาก navbar/dropdown หลังจากผู้ใช้ตั้งใจเข้าใช้งาน ไม่ใช่เนื้อหาหลักของหน้าแรก
12. หน้า login และ register ต้องเป็นประสบการณ์บัญชีลูกค้าของร้านเป็นหลัก ไม่ใส่ตัวเลือกบทบาทหรือข้อความที่ทำให้หน้าเหมือนระบบหลังบ้าน

## 4. ผู้ใช้งานและสิทธิ์
### 4.1 `guest`
ผู้ใช้ทั่วไปที่ยังไม่ได้เข้าสู่ระบบ
- ดูหน้าแรกของร้าน
- ดูเมนูที่พร้อมขาย
- เลือกประเภทการสั่ง
- เพิ่มรายการลงตะกร้า
- ส่งออเดอร์แบบไม่เป็นสมาชิก หากร้านอนุญาต

### 4.2 `customer`
ลูกค้าหรือสมาชิกของร้าน
- ดูเมนูและรายละเอียดเมนู
- เพิ่ม แก้ไข หรือลบรายการในตะกร้าก่อนยืนยันออเดอร์
- ระบุหมายเหตุของแต่ละรายการ
- เลือกทานที่ร้าน กลับบ้าน หรือเดลิเวอรี
- ติดตามสถานะออเดอร์
- ดูประวัติคำสั่งซื้อของตนเอง
- รับใบเสร็จหรือหลักฐานการสั่งซื้อ

### 4.3 `staff`
พนักงานหน้าร้าน
- เปิดออเดอร์ให้ลูกค้า
- เพิ่มหรือแก้ไขรายการก่อนส่งเข้าครัว
- จัดการสถานะโต๊ะ
- ส่งรายการอาหารเข้าครัว
- อัปเดตสถานะเสิร์ฟอาหาร
- ดูออเดอร์ที่เกี่ยวข้องกับการให้บริการ

### 4.4 `kitchen`
พนักงานครัว
- ดูรายการอาหารที่ส่งเข้าครัว
- เห็นจำนวน หมายเหตุ โต๊ะ ประเภทการสั่ง และเวลาเข้า
- เปลี่ยนสถานะอาหาร เช่น รอปรุง กำลังปรุง พร้อมเสิร์ฟ ยกเลิก
- ดูประวัติคิวอาหารตามช่วงเวลา

### 4.5 `cashier`
แคชเชียร์
- ค้นหาออเดอร์
- ตรวจสอบรายการและยอดสุทธิ
- บันทึกการชำระเงิน
- คำนวณเงินทอน
- ออกใบเสร็จ
- ปิดบิล

### 4.6 `manager`
ผู้จัดการร้าน
- ดูแดชบอร์ดและรายงาน
- ตรวจสอบออเดอร์และประวัติการแก้ไข
- อนุมัติการยกเลิกรายการหรือส่วนลดพิเศษ
- ดูประสิทธิภาพการขายและการให้บริการ

### 4.7 `owner`
เจ้าของร้าน
- เข้าถึงภาพรวมการขายทั้งหมด
- ดูรายงาน รายได้ ต้นทุน และประสิทธิภาพร้าน
- เข้าถึงหน้า admin ได้ตามสิทธิ์ที่กำหนด
- ยังสามารถใช้งานหน้าเมนูลูกค้าและสั่งอาหารได้เหมือน customer

### 4.8 `admin`
ผู้ดูแลระบบ
- จัดการผู้ใช้ บทบาท และสิทธิ์
- จัดการเมนู หมวดหมู่ ราคา โต๊ะ โซน และโปรโมชั่น
- จัดการข้อมูลร้าน ภาษี ค่าบริการ และวิธีชำระเงิน
- เข้าถึงรายงานทั้งหมด
- ตรวจสอบและจัดการประวัติการใช้งานของระบบ

## 5. Customer Journey หลัก
1. ลูกค้าเข้าสู่หน้าแรกของร้าน หรือสแกน QR Code ที่โต๊ะ
2. ลูกค้าเห็นภาพรวมร้าน เมนูแนะนำ โปรโมชัน และช่องทางติดต่อ
3. ลูกค้าเข้าสู่หน้าเมนู เลือกหมวดอาหาร ดูรายละเอียด ราคา รูปภาพ และตัวเลือกเพิ่มเติม
4. ลูกค้าเพิ่มรายการลงตะกร้า ระบุจำนวนและหมายเหตุ
5. ลูกค้าตรวจสอบตะกร้า เลือกประเภทการสั่ง และยืนยันออเดอร์
6. ระบบสร้างเลขที่ออเดอร์และส่งรายการที่ต้องปรุงเข้าครัว
7. ลูกค้าหรือพนักงานสามารถดูสถานะออเดอร์ได้
8. ครัวอัปเดตสถานะอาหารจนพร้อมเสิร์ฟ
9. พนักงานเสิร์ฟอาหาร หรือจัดเตรียมสำหรับกลับบ้าน/เดลิเวอรี
10. ลูกค้าชำระเงิน หรือแคชเชียร์รับชำระเงิน
11. ระบบปิดบิลและบันทึกประวัติรายการทั้งหมด

## 7. Functional Requirement
### FR-FULL-00 หน้าแรกเว็บไซต์ร้านอาหารและโครงสร้างการนำทาง
1. `index.html` ต้องเป็นหน้าเว็บไซต์ร้านอาหารสำหรับลูกค้าทั่วไป ไม่ใช่หน้าสารบัญระบบ ไม่ใช่หน้าเอกสาร requirement และไม่ใช่หน้า dashboard หลังบ้าน
2. หน้าแรกต้องสื่อถึงร้านอาหารอย่างชัดเจน เช่น ชื่อร้าน ภาพอาหารหรือบรรยากาศร้าน จุดเด่นของร้าน เมนูแนะนำ โปรโมชัน วิธีสั่งอาหาร รีวิวลูกค้า แกลเลอรี และข้อมูลติดต่อ
3. หน้าแรกต้องไม่ใช้ข้อความอธิบายเชิงระบบ เช่น one port, mockup, POS system, admin system, full-system หรือข้อความชวนกด shortcut ไปหน้าหลังบ้าน
4. Navbar ต้องเป็นชุดเดียวกันทุกหน้า เพื่อให้เว็บเป็นทางเข้าเดียวของร้าน
5. Navbar ฝั่งลูกค้าต้องแสดงเป็นลิงก์ตรง ไม่ทำเป็น dropdown ได้แก่ หน้าแรก เมนู ตะกร้า ติดตาม ประวัติ โปรโมชัน และติดต่อ
6. หน้า sub-page เช่น รายละเอียดเมนูและยืนยันออเดอร์ ไม่ต้องอยู่ใน navbar หลัก แต่ต้องเข้าถึงจากหน้าที่เกี่ยวข้องเท่านั้น เช่น รายละเอียดเมนูเข้าจากหน้าเมนู และยืนยันออเดอร์เข้าจากตะกร้า
7. Navbar ต้องมีปุ่ม `Login` และ `Register` อยู่ข้างกัน โดย `Login` เป็นทางเข้าสำหรับทุกบทบาท และ `Register` เป็นทางสมัครสมาชิกของลูกค้า
8. Navbar ต้องมี dropdown สำหรับ POS, ครัว, แคชเชียร์ และ Admin เพื่อแสดงภาพรวมของระบบใน mockup
9. ในระบบจริง เมนูหลังบ้านต้องถูกควบคุมด้วยสิทธิ์หลัง login แต่ใน mockup สามารถแสดงลิงก์ครบเพื่อให้คณะกรรมการเห็น scope ทั้งระบบ
10. ทุกลิงก์ภายในต้องใช้ root-relative path เช่น `/`, `/auth/login.html`, `/customer/menu.html`, `/admin/menu.html`

### FR-FULL-01 การเข้าสู่ระบบและสิทธิ์
1. ระบบต้องรองรับการเข้าสู่ระบบสำหรับ customer, staff, kitchen, cashier, manager, owner และ admin ผ่านหน้า login เดียว
2. หน้า login ต้องเป็นหน้าบัญชีร้านอาหารสำหรับลูกค้าเป็นหลัก โดยมีฟอร์มอีเมลหรือเบอร์โทร รหัสผ่าน จดจำบัญชี และลิงก์สมัครสมาชิก
3. หน้า login ไม่ต้องมี dropdown เลือกบทบาท เช่น ลูกค้า พนักงาน ครัว แคชเชียร์ หรือหลังร้าน
4. หน้า register ต้องรองรับการสมัครสมาชิกลูกค้า เช่น ชื่อ เบอร์โทร อีเมล รหัสผ่าน ยืนยันรหัสผ่าน ที่อยู่เดลิเวอรี และการยอมรับเงื่อนไข
5. ช่องรหัสผ่านสามารถมีปุ่มไอคอนรูปตาสำหรับแสดงหน้าตา mockup ได้ แต่ห้ามเขียน custom JavaScript เพื่อสลับรหัสผ่าน หากยังจำกัดงานเป็น HTML + Bootstrap CDN
6. ระบบจริงต้องแสดงเมนูและหน้าจอตามสิทธิ์ของผู้ใช้งาน
7. ผู้ใช้ที่ไม่มีสิทธิ์ต้องไม่สามารถเข้าถึงหน้าหรือการทำงานที่ถูกจำกัดได้
8. admin ต้องสามารถเพิ่ม แก้ไข ปิดใช้งานผู้ใช้ และกำหนดบทบาทได้

### FR-FULL-02 หน้าเมนูลูกค้า
1. ลูกค้าต้องสามารถดูหมวดหมู่เมนูและรายการเมนูที่พร้อมขายได้
2. ลูกค้าต้องสามารถดูรูปภาพ ราคา รายละเอียด และสถานะพร้อมขายของเมนูได้
3. เมนูที่ปิดขายต้องแสดงเป็นไม่พร้อมขาย หรือไม่แสดงให้สั่งได้
4. ลูกค้าต้องสามารถค้นหาหรือกรองเมนูตามหมวดหมู่ได้
5. สำหรับ mockup ชุดข้อมูลตัวอย่างต้องมีเมนูครบ 42 รายการ สอดคล้องกับตัวเลขที่หน้าแรกและหน้าเมนูแสดง
6. หน้าเมนูต้องแบ่งหมวดหมู่ 7 หมวด หมวดละ 6 รายการ ได้แก่ อาหารจานเดียว กับข้าว เส้นและแกง ของทานเล่น เครื่องดื่ม ของหวาน และชุดเซ็ต
7. หมวด `ทั้งหมด` ต้องแสดงครบ 42 รายการ และควรมีตัวเลขหรือข้อความช่วยยืนยันจำนวน เช่น `เมนูที่ 01 จาก 42`
8. การเลือกหมวดใน mockup ให้ใช้ Bootstrap tab/pill หรือ component ของ Bootstrap ได้ โดยไม่ต้องเขียน custom JavaScript
9. ปุ่มดูรายละเอียดในแต่ละเมนูต้องเชื่อมไปหน้า `customer/menu-detail.html`

### FR-FULL-03 ตะกร้าสินค้าและการยืนยันออเดอร์
1. ลูกค้าต้องสามารถเพิ่มเมนูหลายรายการลงตะกร้าได้
2. ลูกค้าต้องสามารถแก้ไขจำนวน ลบรายการ และเพิ่มหมายเหตุได้ก่อนยืนยันออเดอร์
3. ระบบต้องคำนวณยอดรวม ส่วนลด ค่าบริการ ภาษี และยอดสุทธิ
4. ระบบต้องตรวจสอบจำนวนอาหารว่ามากกว่า 0
5. ระบบต้องตรวจสอบว่าเมนูยังพร้อมขายก่อนยืนยันออเดอร์

### FR-FULL-04 การสั่งผ่าน QR Code โต๊ะ
1. ลูกค้าต้องสามารถสแกน QR Code เพื่อผูกออเดอร์กับโต๊ะได้
2. ระบบต้องแสดงหมายเลขโต๊ะบนหน้าสั่งอาหาร
3. โต๊ะที่มีออเดอร์เปิดอยู่ต้องแสดงสถานะไม่ว่าง
4. ระบบต้องป้องกันการเปิดออเดอร์ซ้ำสำหรับโต๊ะเดียวกันตามเงื่อนไขของร้าน

### FR-FULL-05 การสั่งแบบกลับบ้านและเดลิเวอรี
1. ลูกค้าหรือพนักงานต้องสามารถเลือกประเภทการสั่งเป็นกลับบ้านหรือเดลิเวอรีได้
2. ออเดอร์เดลิเวอรีต้องรองรับชื่อผู้รับ เบอร์โทร และที่อยู่จัดส่ง
3. ออเดอร์กลับบ้านต้องรองรับเวลาเข้ารับอาหารโดยประมาณ
4. ระบบต้องแสดงประเภทการสั่งให้ครัวและแคชเชียร์เห็นชัดเจน

### FR-FULL-06 การเปิดออเดอร์โดยพนักงาน
1. staff ต้องสามารถเปิดออเดอร์ใหม่ให้ลูกค้าได้
2. ระบบต้องสร้างเลขที่ออเดอร์อัตโนมัติ
3. ออเดอร์ต้องระบุประเภทการสั่ง ผู้รับออเดอร์ โต๊ะ และวันเวลาที่เปิดออเดอร์
4. staff ต้องสามารถเพิ่มรายการอาหารแทนลูกค้าได้

### FR-FULL-07 การแก้ไขและยกเลิกรายการอาหาร
1. ก่อนส่งเข้าครัว ผู้รับออเดอร์ต้องสามารถแก้ไขจำนวนหรือลบรายการได้
2. หลังส่งเข้าครัวแล้ว การแก้ไขหรือลบรายการต้องจำกัดตามสิทธิ์
3. หากยกเลิกรายการ ต้องบันทึกเหตุผล ผู้ทำรายการ และเวลาทำรายการ
4. รายการที่กำลังปรุงหรือพร้อมเสิร์ฟไม่ควรถูกลบโดยผู้ใช้ทั่วไป

### FR-FULL-08 การส่งออเดอร์เข้าครัว
1. ระบบต้องส่งเฉพาะรายการที่ต้องปรุงเข้าหน้าครัว
2. ครัวต้องเห็นเลขที่ออเดอร์ โต๊ะ ประเภทการสั่ง เมนู จำนวน หมายเหตุ และเวลาเข้า
3. ระบบควรระบุเวลาเริ่มทำอาหารและเวลาพร้อมเสิร์ฟได้
4. รายการที่ไม่ต้องปรุง เช่น เครื่องดื่มบางรายการ อาจไม่ต้องเข้าคิวครัวตามที่ admin กำหนด

### FR-FULL-09 การจัดการคิวครัว
1. kitchen ต้องเห็นรายการอาหารแยกตามสถานะ เช่น รอปรุง กำลังปรุง พร้อมเสิร์ฟ ยกเลิก
2. kitchen ต้องสามารถอัปเดตสถานะรายการอาหารได้
3. staff และ cashier ต้องเห็นสถานะล่าสุดของแต่ละรายการ
4. ระบบต้องบันทึกเวลาที่สถานะของรายการอาหารเปลี่ยนแปลง

### FR-FULL-10 การเสิร์ฟและการส่งมอบอาหาร
1. staff ต้องสามารถเปลี่ยนสถานะรายการเป็นเสิร์ฟแล้วได้
2. ออเดอร์กลับบ้านต้องรองรับสถานะเตรียมเสร็จและรับอาหารแล้ว
3. ออเดอร์เดลิเวอรีต้องรองรับสถานะรอจัดส่ง กำลังจัดส่ง และส่งสำเร็จ หากร้านต้องการใช้งาน
4. ระบบต้องแสดงรายการที่พร้อมเสิร์ฟหรือพร้อมส่งมอบอย่างชัดเจน

### FR-FULL-11 การคำนวณราคา
1. ระบบต้องคำนวณยอดรวมจากรายการอาหารจริง
2. ราคาใน order items ต้องเก็บเป็นราคาขณะสั่ง
3. ระบบต้องรองรับส่วนลด ค่าบริการ และภาษีตามที่ร้านกำหนด
4. ยอดสุทธิต้องอัปเดตเมื่อมีการเปลี่ยนแปลงรายการ
5. ระบบต้องแสดงยอดก่อนปิดบิลให้ตรวจสอบได้

### FR-FULL-12 โปรโมชั่นและส่วนลด
1. admin หรือ manager ต้องสามารถสร้างโปรโมชั่นและส่วนลดได้
2. ระบบต้องรองรับส่วนลดแบบจำนวนเงินและเปอร์เซ็นต์
3. ระบบต้องตรวจสอบเงื่อนไขการใช้ส่วนลด เช่น ยอดขั้นต่ำ วันที่ใช้งาน หรือประเภทออเดอร์
4. การให้ส่วนลดพิเศษควรจำกัดสิทธิ์เฉพาะ manager หรือ admin

### FR-FULL-13 การชำระเงิน
1. cashier หรือ staff ต้องสามารถบันทึกการชำระเงินได้
2. ระบบต้องรองรับวิธีชำระเงิน เช่น เงินสด โอน บัตร หรือ QR Payment
3. ระบบต้องคำนวณยอดทอนเมื่อชำระด้วยเงินสด
4. เมื่อชำระครบแล้ว ระบบต้องเปลี่ยนสถานะออเดอร์เป็นชำระแล้วหรือปิดบิลแล้ว
5. ระบบต้องรองรับการชำระเงินหลายช่องทางในออเดอร์เดียว หากร้านกำหนด

### FR-FULL-14 ใบเสร็จและการปิดบิล
1. ระบบต้องสามารถสรุปรายการอาหารทั้งหมดก่อนปิดบิลได้
2. ระบบต้องแสดงยอดรวม ส่วนลด ค่าบริการ ภาษี ยอดสุทธิ เงินรับ และเงินทอน
3. ออเดอร์ที่ปิดบิลแล้วต้องไม่สามารถแก้ไขได้ ยกเว้น admin หรือผู้มีสิทธิ์
4. ระบบควรสามารถแสดงหรือพิมพ์ใบเสร็จได้

### FR-FULL-15 การค้นหาและติดตามออเดอร์
1. ระบบต้องค้นหาออเดอร์ตามเลขที่ออเดอร์ โต๊ะ ประเภทการสั่ง ลูกค้า วันที่ และสถานะได้
2. ระบบต้องกรองออเดอร์ที่ยังไม่ชำระ กำลังปรุง พร้อมเสิร์ฟ หรือปิดบิลแล้วได้
3. ลูกค้าต้องสามารถติดตามสถานะออเดอร์ของตนเองได้
4. staff และ cashier ต้องสามารถดูรายละเอียดออเดอร์ทั้งหมดที่เกี่ยวข้องกับการให้บริการได้

### FR-FULL-16 การจัดการเมนูและหมวดหมู่
1. admin ต้องสามารถเพิ่ม แก้ไข ลบ หรือปิดใช้งานหมวดหมู่เมนูได้
2. admin ต้องสามารถเพิ่ม แก้ไข ลบ หรือปิดขายเมนูได้
3. ต้องกำหนดราคา รูปภาพ รายละเอียด ตัวเลือกเพิ่มเติม และสถานะพร้อมขายได้
4. หมวดหมู่หรือเมนูที่ถูกใช้งานแล้วไม่ควรถูกลบถาวร
5. หน้า admin สำหรับจัดการเมนูต้องใช้ชุดรายการเดียวกับหน้าเมนูลูกค้า เพื่อไม่ให้จำนวนและชื่อเมนูขัดกัน
6. สำหรับ mockup หน้า admin menu ต้องแสดงเมนูครบ 42 รายการ แบ่ง 7 หมวด หมวดละ 6 รายการ
7. ตารางหน้า admin menu ต้องมี pagination 10 รายการต่อหน้า
8. Pagination ต้องมีทั้งด้านบนและด้านล่างของตาราง
9. Pagination ใน mockup ใช้ Bootstrap tab/pill หรือ pagination component ได้ โดยไม่ต้องเขียน custom JavaScript

### FR-FULL-17 การจัดการโต๊ะและโซน
1. admin ต้องสามารถเพิ่ม แก้ไข ปิดใช้งานโต๊ะและโซนได้
2. ระบบต้องรองรับจำนวนที่นั่งและสถานะโต๊ะ
3. ระบบต้องรองรับ QR Code ประจำโต๊ะ
4. โต๊ะที่มีออเดอร์เปิดอยู่ต้องแสดงสถานะไม่ว่าง

### FR-FULL-18 การจัดการข้อมูลร้าน
1. admin ต้องสามารถแก้ไขชื่อร้าน โลโก้ ที่อยู่ เบอร์โทร และเวลาเปิดปิดได้
2. admin ต้องสามารถตั้งค่าภาษี ค่าบริการ และรูปแบบเลขที่ออเดอร์ได้
3. admin ต้องสามารถตั้งค่าวิธีชำระเงินที่ร้านรองรับได้

### FR-FULL-19 รายงานและแดชบอร์ด
1. ระบบต้องมีแดชบอร์ดสรุปยอดขาย ออเดอร์เปิดอยู่ คิวครัว และสถานะโต๊ะ
2. ระบบต้องมีรายงานยอดขายรายวัน รายเมนู และรายประเภทการสั่ง
3. ระบบต้องมีรายงานเมนูขายดี จำนวนออเดอร์ และยอดขายตามช่วงเวลา
4. ระบบต้องมีรายงานยอดขายแยกตามวิธีชำระเงิน
5. ระบบต้องมีรายงานออเดอร์ที่ยกเลิกหรือแก้ไข
6. ระบบควรมีรายงานประสิทธิภาพครัว เช่น เวลาเฉลี่ยในการปรุงอาหาร

### FR-FULL-20 ประวัติการใช้งาน
1. ระบบต้องบันทึกประวัติการเปิดออเดอร์
2. ระบบต้องบันทึกประวัติการเพิ่ม แก้ไข และยกเลิกรายการอาหาร
3. ระบบต้องบันทึกประวัติการส่งเข้าครัวและเปลี่ยนสถานะอาหาร
4. ระบบต้องบันทึกประวัติการชำระเงินและปิดบิล
5. admin และ manager ต้องสามารถตรวจสอบประวัติการใช้งานได้

## 8. Business Rule
1. เมนูที่ปิดขายต้องไม่สามารถเพิ่มลงออเดอร์ใหม่ได้
2. ราคาในรายการอาหารของออเดอร์ต้องเก็บเป็นราคาขณะสั่ง
3. โต๊ะ 1 โต๊ะไม่ควรมีออเดอร์เปิดซ้ำหลายใบพร้อมกัน เว้นแต่ร้านกำหนดให้รองรับ
4. ออเดอร์ที่ปิดบิลแล้วต้องเป็นอ่านอย่างเดียว ยกเว้นผู้มีสิทธิ์ระดับสูง
5. รายการอาหารที่ส่งเข้าครัวแล้วต้องมีข้อจำกัดในการแก้ไขหรือลบ
6. การยกเลิกรายการหลังส่งเข้าครัวต้องบันทึกเหตุผลเสมอ
7. การให้ส่วนลดเกินเงื่อนไขปกติต้องใช้สิทธิ์ manager หรือ admin
8. ระบบต้องไม่ส่งรายการที่ไม่ต้องปรุงเข้าครัว หากเมนูนั้นถูกตั้งค่าไว้เช่นนั้น
9. ออเดอร์เดลิเวอรีต้องมีข้อมูลติดต่อหรือที่อยู่ครบถ้วนก่อนยืนยัน
10. การปิดบิลต้องเกิดขึ้นหลังตรวจสอบยอดชำระเงินครบถ้วนแล้ว

## 9. Validation และความปลอดภัย
1. ต้องตรวจสอบจำนวนอาหารว่ามากกว่า 0
2. ต้องตรวจสอบราคา ส่วนลด ภาษี และยอดสุทธิให้ถูกต้องก่อนบันทึก
3. ต้องตรวจสอบว่าเมนูยังพร้อมขายก่อนยืนยันออเดอร์
4. ต้องตรวจสอบข้อมูลการชำระเงินก่อนปิดบิล
5. ต้องกำหนดสิทธิ์แยกหน้าลูกค้า หน้ารับออเดอร์ หน้าครัว หน้าแคชเชียร์ หน้ารายงาน และหน้า admin
6. ต้องป้องกันผู้ใช้เข้าถึงข้อมูลหรือการทำงานที่เกินสิทธิ์
7. รหัสผ่านต้องไม่เก็บเป็นข้อความธรรมดาหากพัฒนาระบบจริง
8. ข้อมูลสำคัญ เช่น การแก้ไขราคา การยกเลิกรายการ และการปิดบิล ต้องถูกบันทึกใน audit log

## 10. Non-Functional Requirement
1. ระบบควรรองรับการใช้งานบนมือถือ แท็บเล็ต และเดสก์ท็อป
2. หน้าลูกค้าควรใช้งานง่ายและโหลดเร็ว
3. หน้าแรกควรดูเป็นเว็บไซต์ร้านอาหารจริง มีภาพและข้อความที่เหมาะกับลูกค้า
4. หน้าครัวควรอ่านง่าย เห็นคิวชัด และใช้งานได้บนจอขนาดใหญ่
5. หน้าพนักงานและแคชเชียร์ควรเหมาะกับการใช้งานซ้ำบ่อยและทำงานเร็ว
6. ระบบควรออกแบบให้แยกสิทธิ์และขยายฟีเจอร์ได้ในอนาคต
7. หากทำเป็น mockup HTML ให้ใช้ข้อมูลตัวอย่างแทนการเชื่อมต่อฐานข้อมูล

## 15. ขอบเขตสำหรับ Mockup HTML
1. ใช้ HTML และ Bootstrap 5 CDN ได้
2. ใช้ Bootstrap JavaScript CDN ได้เฉพาะ component ของ Bootstrap เช่น navbar collapse, dropdown, tabs, pills, modal หรือ pagination แบบ tab
3. ไม่ต้องมีระบบฐานข้อมูลจริง
4. ไม่ต้องมี Node.js, PHP หรือ backend
5. ข้อมูลทั้งหมดสามารถเป็นข้อมูลตัวอย่าง
6. ปุ่มและฟอร์มสามารถเป็น mockup เพื่อแสดงหน้าตาและ flow ของระบบ
7. `index.html` ต้องเป็นหน้าเว็บร้านสำหรับลูกค้าทั่วไป ไม่ใช่หน้าสารบัญหรือเอกสารระบบ
8. ทุกหน้าต้องใช้ navbar ชุดเดียวกัน เพื่อให้ระบบเป็น one port เดียว ไม่แยก public และ back office ด้วย navbar คนละชุด
9. Navbar ต้องมีลิงก์ฝั่งลูกค้าเป็น nav link โดยตรง ได้แก่ หน้าแรก เมนู ตะกร้า ติดตาม ประวัติ โปรโมชัน และติดต่อ
10. Navbar ต้องไม่ใส่ sub-page เช่น รายละเอียดเมนู และยืนยันออเดอร์ เป็นลิงก์หลัก
11. Navbar ต้องมี `Login` และ `Register` อยู่ข้างกัน
12. `Login` ต้องเป็นจุดเข้าสู่ระบบเดียวสำหรับ customer, staff, kitchen, cashier, manager, owner และ admin
13. ผู้ใช้ทุกบทบาทต้องสามารถเข้าหน้าเมนูลูกค้าและสั่งอาหารผ่าน flow เดียวกันได้ เช่น เจ้าของร้านหรือพนักงาน login แล้วก็ยังสั่งอาหารกินเองได้
14. ในระบบจริงสิทธิ์ของ role จะใช้ควบคุมการใช้งานจริงของแต่ละเมนู แต่ใน mockup ให้แสดงลิงก์ครบเพื่อเห็นภาพรวมของระบบ
15. ใช้ Font Awesome CDN ร่วมกับ Bootstrap สำหรับไอคอนใน navbar ปุ่ม และส่วนควบคุมหลัก
16. ห้ามสร้างไฟล์ CSS หรือ JavaScript แยก และห้ามเขียน custom JavaScript ในไฟล์ HTML
17. ห้ามใช้ `<base href>` ในไฟล์ HTML ให้ใช้ root-relative path สำหรับลิงก์ภายใน เช่น `/`, `/auth/login.html`, `/customer/menu.html`, `/cashier/search.html`
18. ห้ามทำหน้าแรกให้เป็น shortcut หรือ link hub สำหรับเข้าระบบต่าง ๆ เนื้อหาหลักของหน้าแรกต้องเป็นร้านอาหาร
19. หน้าเมนูลูกค้าและหน้า admin menu ต้องใช้ชุดเมนูตัวอย่างเดียวกัน 42 รายการ เพื่อให้จำนวนและชื่อเมนูไม่ขัดกัน
20. หน้า admin menu ต้องมี pagination 10 รายการต่อหน้า และมี pagination ทั้งบนและล่างของตาราง
21. แยกไฟล์ mockup ตามโฟลเดอร์ของประเภทงาน เพื่อให้อ่านง่ายและใกล้เคียงระบบจริง
    - `auth/` สำหรับหน้า login และ register
    - `customer/` สำหรับหน้าเมนู ตะกร้า checkout สถานะออเดอร์ และประวัติลูกค้า
    - `pos/` สำหรับหน้า POS ของพนักงานหน้าร้าน
    - `kitchen/` สำหรับหน้าครัว
    - `cashier/` สำหรับหน้าชำระเงินและใบเสร็จ
    - `admin/` สำหรับหน้า back office และรายงาน

## 16. Acceptance Criteria สำหรับ Mockup HTML
1. เปิด `/` แล้วต้องเห็นหน้าเว็บร้านอาหาร ไม่ใช่เอกสาร ไม่ใช่ dashboard และไม่ใช่หน้า shortcut เข้าระบบ
2. หน้าแรกต้องมีอย่างน้อย hero, เมนูแนะนำวันนี้, โปรโมชันสำหรับลูกค้า, เรื่องราวหรือจุดเด่นของร้าน, วิธีสั่งอาหาร, รีวิวหรือบรรยากาศร้าน และ footer ที่ดูเป็นมืออาชีพ
3. รูปภาพอาหารหรือบรรยากาศร้านต้องมีในหน้าแรก หน้าเมนูลูกค้า และหน้ารายละเอียดเมนู
4. Navbar ทุกหน้าต้องเหมือนกัน และต้องมี `Login` กับ `Register` อยู่ข้างกัน
5. ลิงก์ลูกค้าใน navbar ต้องเป็นลิงก์ตรง ไม่อยู่ใน dropdown
6. `customer/menu-detail.html` ต้องไม่อยู่ใน navbar และต้องเข้าจากปุ่มดูรายละเอียดในหน้าเมนูเท่านั้น
7. `customer/checkout.html` ต้องไม่อยู่ใน navbar และต้องเข้าจากตะกร้าเท่านั้น
8. หน้า login ต้องไม่มีตัวเลือก `เข้าใช้งานในฐานะ`
9. หน้า login และ register ต้องไม่มีข้อความที่ทำให้ดูเป็นหน้า shortcut ของทีมร้านหรือระบบหลังบ้าน
10. หน้า `customer/menu.html` ต้องแสดงเมนูครบ 42 รายการในหมวดทั้งหมด และแต่ละหมวดต้องมี 6 รายการ
11. หน้า `admin/menu.html` ต้องแสดงเมนูครบ 42 รายการชุดเดียวกับหน้า customer menu
12. หน้า `admin/menu.html` ต้องแบ่ง pagination 10 รายการต่อหน้า จำนวน 5 หน้า และมีปุ่ม pagination ทั้งด้านบนและด้านล่างตาราง
13. ทุกฟอร์มต้องมี label หรือ accessible name ครบถ้วน
14. รูปภาพทุกภาพต้องมี `alt`
15. ห้ามมี `<base href>`
16. ห้ามมีไฟล์ `.css` หรือ `.js` ที่สร้างขึ้นเอง
17. ห้ามมี custom inline JavaScript ยกเว้น Bootstrap JavaScript CDN
18. Link ภายในต้องใช้ root-relative path เสมอ

## 18. Development Phases สำหรับการพัฒนาระบบจริง
การทำงานจริงควรแบ่งเป็น phase เพื่อไม่ให้ขอบเขตกระโดดจาก HTML mockup ไปเป็นระบบใหญ่ทันที

### Phase 0: Requirement, UX Direction และ Scope Confirmation
เป้าหมาย: ยืนยันภาพรวมร้าน หน้าจอหลัก flow การสั่งอาหาร สิทธิ์ผู้ใช้ และขอบเขตงานก่อนลงมือพัฒนา

Deliverable:
- Requirement ฉบับชัดเจน
- Sitemap และ navigation structure
- User journey ของลูกค้า พนักงาน ครัว แคชเชียร์ และ admin
- รายการหน้าที่ต้องทำทั้งหมด
- ชุดข้อมูลตัวอย่าง เช่น เมนู 42 รายการ หมวดหมู่ โต๊ะ และสถานะออเดอร์

### Phase 1: Static HTML Mockup
เป้าหมาย: ทำหน้าตาและ flow ของระบบด้วย HTML, Bootstrap 5 CDN, Bootstrap JS CDN และ Font Awesome CDN เท่านั้น

ขอบเขต:
- หน้าแรกต้องเป็นเว็บไซต์ร้านอาหาร
- หน้าเมนูลูกค้า ตะกร้า checkout สถานะ และประวัติ
- หน้า login และ register
- หน้า POS, kitchen, cashier และ admin
- Navbar และ footer ชุดเดียวกันทุกหน้า
- ไม่มี database, backend, Node.js, PHP, custom CSS หรือ custom JavaScript

Deliverable:
- ไฟล์ HTML แยกตามโฟลเดอร์ `auth/`, `customer/`, `pos/`, `kitchen/`, `cashier/`, `admin/`
- Mockup พร้อมตรวจ usability, responsive, accessibility และความครบของ flow

### Phase 2: Backend Foundation, Database และ API
เป้าหมาย: สร้างฐานระบบจริงให้รองรับข้อมูล ผู้ใช้ สิทธิ์ และ API ก่อนเริ่มทำ business flow เต็มรูปแบบ

ขอบเขต:
- เลือก tech stack สำหรับ backend และ database
- ออกแบบ database schema จากตารางขั้นต่ำ เช่น `users`, `roles`, `permissions`, `customers`, `menu_items`, `orders`, `order_items`, `payments`, `audit_logs`
- ทำ migration และ seed data สำหรับเมนู 42 รายการ หมวดหมู่ โต๊ะ บทบาท และผู้ใช้ตัวอย่าง
- ทำระบบ authentication และ authorization
- ทำ role-based access control สำหรับ customer, staff, kitchen, cashier, manager, owner และ admin
- ทำ API พื้นฐานสำหรับ login, register, profile, menu, category, table, order, payment และ report
- ทำ validation กลาง เช่น ราคา จำนวน สถานะเมนู สิทธิ์ และข้อมูลชำระเงิน
- ทำ audit log สำหรับ action สำคัญ

Deliverable:
- Database schema และ migration
- Seed data
- API documentation
- Authentication และ RBAC พร้อมใช้งาน
- Backend พร้อมเชื่อมกับ frontend ใน phase ถัดไป

### Phase 3: Customer Ordering และ POS Core
เป้าหมาย: ทำให้ลูกค้าและพนักงานหน้าร้านสร้างออเดอร์จริงได้

ขอบเขต:
- โหลดเมนูจริงจากฐานข้อมูล
- ตะกร้าสินค้า เพิ่มรายการ แก้ไขจำนวน ลบรายการ และหมายเหตุ
- ยืนยันออเดอร์
- สร้างเลขที่ออเดอร์อัตโนมัติ
- รองรับประเภทการสั่ง ทานที่ร้าน กลับบ้าน และเดลิเวอรี
- รองรับ QR Code โต๊ะ
- POS เปิดออเดอร์แทนลูกค้า
- จัดการโต๊ะและสถานะโต๊ะ
- คำนวณยอดรวม ส่วนลด ค่าบริการ ภาษี และยอดสุทธิ

Deliverable:
- Customer ordering flow ใช้งานได้จริง
- POS core ใช้งานได้จริง
- ออเดอร์ถูกบันทึกลงฐานข้อมูลและพร้อมส่งเข้าครัว

### Phase 4: Kitchen, Cashier, Payment และ Receipt
เป้าหมาย: ทำ order lifecycle ตั้งแต่ส่งเข้าครัวจนปิดบิล

ขอบเขต:
- ส่งรายการที่ต้องปรุงเข้าคิวครัว
- หน้าครัวแสดงคิว รอปรุง กำลังปรุง พร้อมเสิร์ฟ และประวัติ
- staff เห็นรายการพร้อมเสิร์ฟและเปลี่ยนสถานะเสิร์ฟแล้วได้
- cashier ค้นหาออเดอร์ ตรวจยอด รับชำระเงิน และปิดบิล
- รองรับวิธีชำระเงิน เช่น เงินสด โอน บัตร หรือ QR Payment ตามที่ร้านกำหนด
- คำนวณเงินทอน
- ออกใบเสร็จ
- บันทึก payment logs, kitchen logs และ order logs

Deliverable:
- Kitchen display ใช้งานได้จริง
- Cashier/payment flow ใช้งานได้จริง
- Receipt และปิดบิลได้จริง

### Phase 5: Admin, Back Office, Reports และ Settings
เป้าหมาย: ให้เจ้าของร้านจัดการข้อมูลและดูรายงานได้ครบ

ขอบเขต:
- จัดการผู้ใช้ บทบาท และสิทธิ์
- จัดการข้อมูลร้าน ภาษี ค่าบริการ เวลาเปิดปิด และวิธีชำระเงิน
- จัดการหมวดหมู่ เมนู ตัวเลือกเมนู รูปภาพ ราคา และสถานะพร้อมขาย
- จัดการโต๊ะ โซน และ QR Code
- จัดการโปรโมชั่นและส่วนลด
- ค้นหาออเดอร์และดูประวัติการแก้ไข
- รายงานยอดขาย รายเมนู รายช่วงเวลา รายวิธีชำระเงิน รายสถานะออเดอร์ รายเมนูขายดี และประสิทธิภาพครัว
- ตารางหลังบ้านควรมี search, filter, sort และ pagination

Deliverable:
- Back office ใช้งานได้ครบสำหรับเจ้าของร้านและ admin
- Report dashboard พร้อมข้อมูลจากระบบจริง

### Phase 6: QA, Security, Performance และ Deployment
เป้าหมาย: ทำให้ระบบพร้อมใช้งานจริงอย่างปลอดภัยและเสถียร

ขอบเขต:
- Functional test ตาม requirement
- Role permission test
- Accessibility test เช่น label, alt, keyboard navigation
- Responsive test บนมือถือ แท็บเล็ต และเดสก์ท็อป
- Security test เช่น password hashing, session, CSRF, XSS, SQL injection และ access control
- Performance test สำหรับหน้าเมนู คิวครัว และรายงาน
- Backup และ restore plan
- Deploy production
- Logging, monitoring และ error tracking
- คู่มือใช้งานสำหรับ staff, kitchen, cashier และ admin

Deliverable:
- Production release
- Test report
- Deployment document
- User manual

### Phase 7: Enhancement และ Operation หลังเปิดใช้งาน
เป้าหมาย: เพิ่มความสามารถหลังระบบหลักใช้งานได้แล้ว

ตัวอย่างงาน:
- ระบบสมาชิก คะแนนสะสม และคูปอง
- ระบบสต็อกวัตถุดิบ
- เชื่อมต่อ delivery platform
- เชื่อมต่อ payment gateway หรือ QR Payment จริง
- เชื่อมต่อเครื่องพิมพ์ครัวและเครื่องพิมพ์ใบเสร็จ
- Real-time notification สำหรับครัวและพนักงาน
- Advanced analytics สำหรับเมนูขายดี ช่วงเวลาขายดี และต้นทุน
- Multi-branch หากร้านมีหลายสาขา

สรุป phase:
- ถ้านับตั้งแต่การวางแผน จะมี 8 phase คือ Phase 0 ถึง Phase 7
- ถ้านับเฉพาะงานพัฒนาหลัง requirement จะมี 7 phase คือ Phase 1 ถึง Phase 7
- Phase ปัจจุบันของงาน HTML นี้คือ Phase 1
- Phase ถัดไปที่ควรทำถ้าจะพัฒนาเป็นระบบจริงคือ Phase 2: Backend Foundation, Database และ API
